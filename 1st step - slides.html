1st Step

Learning Code Through the Fortress Project

A macro-to-micro guide for complete beginners

7 levels: from "what is software?" to watching a live factory

Level 1: What Is Software?

Fortress is made of 4 layers that each do a different job:

Frontend

What the user sees and clicks

Vanilla JS SPA (Single Page App)

Dashboard, progress bars, export buttons

Backend API

The traffic cop between user and system

Python + FastAPI

Login, start jobs, serve data

Pipeline

The factory worker that finds data

Python (background process)

Scrapes Maps, crawls websites

Database

The memory that stores everything

PostgreSQL

14.7 million French companies

Level 1: How the 4 Layers Communicate

Think of a restaurant with a dining room, a waiter, a kitchen, and a pantry:

User clicks "Search bakeries in Paris" | [ FRONTEND ] ---request---> [ BACKEND API ] ---"start job"---> [ PIPELINE ] (dining room) (waiter) (kitchen) | | | | <---"here's the | scrapes Google Maps | | latest data"--- | crawls websites | | | | Shows results Serves data [ DATABASE ] live on screen back and forth (the pantry) without reloading 14.7M companies stored

Key insight: The Pipeline runs separately from the Backend API. The API handles user requests. The Pipeline does heavy scraping in the background. They don't block each other.

Level 2: Why Split Code Into Files?

Imagine a company where HR, sales, and accounting share one giant notebook. Chaos.

The Problem

  • One file with 10,000 lines = impossible to navigate
  • Two people editing the same file = conflicts
  • Bug in the scraper breaks the login page
  • Can't find anything when things go wrong

The Solution

  • Split by responsibility
  • Each file does one job
  • Files talk to each other via import
  • Fix one "department" without touching others

Level 2: Fortress Folder Map

fortress/ config/ Manager's Rulebook (settings, passwords) api/ Front Desk (web server, login, routes) database/ Blueprint Room (table definitions) module_a/ Sorting Hat (search interpreter, triage) module_c/ Field Workers (Maps scraper, website crawler) module_d/ Assembly Line (enricher, batch processor) frontend/ Storefront (SPA dashboard UI) data/ Storage Room (checkpoints, logs, exports) runner.py Factory Manager (orchestrates the pipeline)

Notice: There's no module_b/ — the modules jump from a to c to d. Each module is a team: module_a decides what to scrape, module_c does the scraping, module_d processes and saves results.

Level 3: The 5-Stage Pipeline

When a user clicks "Search," Fortress runs this assembly line — each stage hands off to the next:

1. Interpret

Translates "bakeries in Paris" into a database query

→

2. Triage

Sorts companies into buckets: reuse, skip, or scrape

→

3. Enrich

Scrapes Maps & websites for phone, email, socials

→

4. Wave Process

Saves in batches of 50 with crash-recovery checkpoints

→

5. Complete

Cleanup, close browsers, mark job as done

The MVP Phone Gate: During Stage 3, if no phone number is found after all sources are tried, the company is rejected and replaced. A phone number is the minimum requirement.

Level 4: Variables & Conditionals

Variables = Labeled Boxes

Store information the program needs to remember

batch_size = 50 # companies per wave wave_current = 3 # which batch now company_name = "GEODIS" # name to look up phone = None # empty box (not found yet)

Conditionals = If/Then Decisions

The program makes choices based on rules

if phone is not None: # Phone found! Company is qualified save_to_database(company) else: # No phone = reject and replace replace_with_next_company()

This is the "MVP Phone Gate" — Fortress's most important decision

Level 4: Loops & Functions

Loops = Repeat Until Done

"Do this for every company in the batch"

for company in wave_companies: data = scrape_maps(company) if data.phone: save(data) # Keep it else: replace(company) # Swap it # After 50 companies: save checkpoint!

Functions = Reusable Mini-Programs

Write once, call by name whenever needed

def _assess_match(maps_name, registry_name): """Is this the right business?""" if names_match and cities_match: return "high" # Trust it elif names_match: return "low" # Maybe branch else: return "none" # Wrong business

Level 4: Classes & Imports

Classes = Blueprints for Complex Tools

A function is a verb. A class is a noun with memory.

class CurlClient: timeout = 8 # Max 8s per page delay = 0.4 # Wait between visits impersonate = "chrome" def get(self, url): # Visit webpage using these rules ...

The CurlClient remembers its settings across every page it visits

Imports = Files Talking to Each Other

One file borrows code from another

# Inside runner.py (the boss): from fortress.module_a import interpret from fortress.module_a import triage from fortress.module_d import enricher # runner.py doesn't know HOW to scrape # It delegates to specialist files

Each file focuses on one job. The boss just coordinates.

Level 4: Async/Await (Multitasking)

Normal code: waits for Google Maps to load → entire system freezes

Async code: starts the request, does other work while waiting → keeps serving users

async def enrich_company(company): data = await scrape_google_maps(company) # Start scraping, don't freeze! # While Maps loads (3 seconds), the server can: # - Update the user's dashboard # - Answer another user's request # - Process other database queries await save_to_database(data) # Save when Maps responds

Why Fortress needs this: The Pipeline talks to Google Maps, crawls websites, and queries a 14.7M-row database. The Backend API must keep answering the frontend's requests while all of this happens. Without async, every network request would freeze everything.

Level 5: Database = Organized Filing Cabinet

Table

One spreadsheet tab. Holds one type of thing. "Companies" table, "Contacts" table, etc.

Column

A category header: Name, Phone, City. Defines what kind of data each position holds.

Row

One entry. One specific company. 14.7 million rows = 14.7 million companies.

Primary Key

A unique ID for each row. Two bakeries named "Paul"? The SIREN number tells them apart.

Foreign Key = a link connecting tables. The contacts table stores the company's siren so the system knows "this phone number belongs to company #123456789."

Level 5: Fortress Database Tables

Key rule: The companies table's identity fields (name, address, SIREN) are protected — the pipeline never changes who a company is. All enriched data (phone, email) goes into contacts, linked by SIREN.

Level 6: Design Patterns

Checkpoint / Resume

Save progress to disk after every 50 companies. Crash? Read the file, resume at wave N+1. Zero lost work.

Seen Set

A list of already-processed company IDs in the checkpoint. On restart, skip anything already done. No duplicates.

Data Bank (Smart Reuse)

Already scraped a company last week? Skip scraping, instantly reuse. Massive time savings.

Waterfall + Fast-Fail

Try every data source in order. If none find a phone: reject immediately. Don't waste time on dead ends.

Rate Limiting

One Maps search at a time. Random delays. Chrome impersonation. Act human, don't get banned.

Level 7: The Live Factory

This is where everything connects. Three things happen simultaneously:

1. Immediate Save

The Kitchen Worker

The moment a company passes the Phone Gate, it is saved to the database instantly — not after the whole batch finishes. This is the on_save rule.

2. Polling

The Waiter Checking the Kitchen

The frontend sends a request to the backend every 1.5 seconds, asking: "Any new companies yet?" This is a timer (setInterval) running in JavaScript.

3. DOM Patching

The Assembly Line Display

When the backend says "yes, 1 new bakery," the frontend doesn't reload the page. It surgically slides the new company card onto the screen and ticks up the progress counter.

Level 7: Why This Ties Everything Together

The Live Factory is why every earlier concept exists:

ConceptWithout ItWith It
Async (Level 4)Backend freezes while Pipeline scrapes — can't answer the frontend's pollsBackend serves polls while Pipeline scrapes in parallel
SPA (Level 6)Every update reloads the whole page — white flash, scroll reset, terrible UXNew cards slide in smoothly, counters animate, no reload
on_save (Level 6)User waits for entire wave of 50 before seeing anythingCompanies appear one-by-one, like an assembly line
4 Layers (Level 1)Pipeline inside the API = one slow request blocks everythingPipeline runs separately — API stays fast for the user

The user sees companies sliding onto the screen in real time — and now you understand every piece of code that makes that happen.
