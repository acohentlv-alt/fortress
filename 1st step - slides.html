Level 1: The Golden Rule of the Restaurant

The Pipeline runs as a separate background process from the Backend API. Why?

Imagine if the Waiter had to personally drive to Google Maps to find a phone number. They'd be gone for 10 seconds — and the entire restaurant would freeze. Because the Waiter and Kitchen work independently, the Waiter never stops serving tables while the Kitchen is cooking.

1. You click "Search transport 66" 2. FRONTEND sends request to BACKEND API 3. API logs the order and tells the PIPELINE to start 4. Pipeline scrapes Google Maps + company websites 5. The moment a phone is found → saved to DATABASE instantly 6. Frontend asks the API every 1.5 seconds: "Any new data?" 7. New company cards slide onto screen in real time

Level 2: Why Not One Big File?

Imagine a massive company where HR, Sales, Accounting, and Security all share a single million-page notebook. If a security guard needs to update a password, they flip through thousands of sales records just to find the right spot. Chaos!

The Problem

  • One file with 10,000 lines = impossible to navigate
  • Two people editing the same file = conflicts
  • Bug in the scraper breaks the login page
  • Can't find anything when things go wrong

The Solution

  • Independence: fix scraper without breaking login
  • Teamwork: multiple devs work in parallel
  • Readability: know exactly where to look
  • Reusability: curl_client.py used by any file that needs to visit a webpage

Level 4: Variables & Conditionals

Variables = Labeled Boxes

Scoreboards that hold numbers and text that change over time.

batch_size = 50 # total companies requested wave_current = 3 # which chunk we're on now company_name = "GEODIS" # name to look up phone = None # empty box (not found yet)

The pipeline constantly updates wave_current so the frontend can display a live progress bar.

Conditionals = If/Then Decisions

A strict bouncer checking IDs at the door.

if phone is not None: # Phone found! Company is qualified save_to_database(company) else: # No phone = reject and replace replace_with_next_company()

This is the "MVP Phone Gate" — Fortress's most important decision. It guarantees the user only gets usable data.

Level 4: Loops & Functions

Loops = Assembly Line Worker

Repeating the exact same actions until the daily quota is met.

for company in wave_companies: data = scrape_maps(company) if data.phone: save(data) # Keep it else: replace(company) # Swap it # After 50 companies: save checkpoint! # Power goes out? Resume right here.

The wave loop breaks a massive job into safe, bite-sized pieces.

Functions = Reusable Mini-Programs

A specific action verb that takes inputs and hands back an answer.

def _assess_match(maps_name, registry_name): """Is this the right business?""" if names_match and cities_match: return "high" # Trust it elif names_match: return "low" # Maybe branch else: return "none" # Wrong business

Centralizes logic so the system safely decides: keep the phone number or discard it to prevent a false positive.

Level 4: Classes & Imports

Classes = Machine Blueprints

A function is a verb. A class is a noun — a complex machine with its own dials, settings, and memory.

class CurlClient: timeout = 8 # Max 8s per page delay = 0.4 # Wait between visits impersonate = "chrome" def get(self, url): # Visit webpage using these rules ...

CurlClient remembers its settings across every page. It carries a cryptographic disguise to sneak past bot detection.

Imports = Calling Another Department

Picking up the phone and asking a specialized department for help.

# Inside runner.py (the boss): from fortress.module_a import interpret from fortress.module_a import triage from fortress.module_d import enricher # runner.py doesn't know HOW to scrape # It delegates to specialist files

Each file focuses on one job. The boss just coordinates — no copy-pasting code everywhere.

Level 4: Async/Await (Multitasking)

Analogy: Starting the laundry and walking away to wash the dishes while it runs — instead of staring blankly at the washing machine for an hour.

Normal code: waits for Google Maps to load → entire system freezes for 3 seconds

Async code: starts the request, does other work while waiting → keeps serving users

async def enrich_company(company): data = await scrape_google_maps(company) # Start scraping, don't freeze! # While Maps loads (3 seconds), the server can: # - Answer the frontend's "any new data?" poll # - Update another user's dashboard # - Process other database queries await save_to_database(data) # Save when Maps responds

Why Fortress needs this: Web scraping is incredibly slow. Async ensures the Backend API (the waiter) can continuously answer the frontend's requests while the Pipeline (the kitchen) does the heavy lifting in the background.

Level 5: Database = A Giant Filing Cabinet

Instead of throwing everything into one giant Word document, programmers use a database to keep millions of pieces of information instantly searchable.

Table

A single drawer in the filing cabinet. Only holds one type of thing — you wouldn't mix company addresses with pipeline progress reports.

Column

The category labels inside the drawer: Name, Phone, City. Defines what kind of data each position holds.

Row

One actual entry. One specific company. 14.7 million rows = 14.7 million companies.

Primary Key

A unique ID that guarantees no two rows are confused. Two bakeries named "Paul"? The SIREN number tells them apart.

Foreign Key = a digital bridge connecting two tables. The contacts table stores the company's siren so the system knows "this phone number belongs to company #123456789."

The rule: The pipeline is never allowed to modify the identity fields of companies (name, address, SIREN). All enriched data (phone, email, website) goes into contacts, linked by the company's SIREN. Think of query_tags as sticky notes — a single company can have multiple tags if found in different searches.

Level 6: Design Patterns (Smart Business Rules)

Checkpoint / Resume

After every 50 companies, save a physical bookmark file to disk. Power goes out? Read the file, resume at wave N+1. Zero lost work.

Seen Set

The checkpoint includes a list of every SIREN already processed. On restart, skip anything already done. No accidental duplicates.

Data Bank (Smart Reuse)

Company scraped last week? Triage tags it GREEN — instantly reuse, skip scraping. Massive time savings.

Waterfall + Fast-Fail

Try data sources in strict order: Maps first, then website crawl. If none find a phone: reject immediately. Don't waste time on dead ends.

Rate Limiting (Be Polite)

100 pages/second = banned IP. asyncio.Lock() forces one Maps search at a time. Random 2-3s delays. Chrome impersonation. Act human.

Level 7: The Live Factory

How does invisible backend code make the progress bar move on your screen in real time? Three processes fire simultaneously:

1. Immediate Save

The Kitchen Worker

The pipeline doesn't wait to finish baking all 50 cookies to tell you. The exact millisecond a company passes the Phone Gate, on_save fires and permanently saves that single company to the database.

2. Polling

The Heartbeat

Your browser doesn't wait for the server to call it. The frontend JavaScript runs a timer (setInterval) that asks the backend every 1.5 seconds: "Do you have any new data yet?"

3. DOM Patching

The Magic Trick

When the backend says "yes, 1 new bakery!" — the frontend doesn't redraw the whole screen. It surgically slides the new company card onto the screen and ticks the progress counter up by one. No page reload.
